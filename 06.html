Logical Design

TOP        

A logical design transforms a conceptual schema, into a set of relational schemas. Transformation of a conceptual schema is controlled by a collection of simple rules replacing the components of a simplified class diagram with linear specifications relational schemas, i.e. schema-name( attribute, attribute, ... ) primary key = ... foreign key = ...


Application of the transformation rules is divided into few steps each one replacing one type of the components of a conceptual schema.


A logical design is performed in the following steps:


First, the multivalued attributes are replaced with the classes of objects, and depending on the semantics of multivalued attributes, either with one to many, or with many to many associations.


In the second step, the association classes and link attributes, are replaced with the triples that consists of: one to many association, class-of-objects, and many-to-one-association.


In the third step, many-to-many associations, are replaced with the triples that consists of: one to many association, class of objects, many to one association.


In the fourth step, the qualifications are replaced with one to many associations, and with composite identifiers in object classes on "many" side of one to many associations.


The remaining steps are listed on the next slide.


Methodology

  • The logical design transforms a conceptual schema into a set of relational schemas
  • The logical design is performed as a sequence of the following steps:
  • (1) The multivalued attributes are replaced with the classes of objects and depending on the semantics of multivalued attribute either with one-to-many or many-to-many associations
  • (2) The associaton classes and link attributes are replaced with the triples (one-to-many association:class-of-objects :many-to-one-association)
  • (3) Many-to-many associations are replaced with triples (one-to-many association:class-of-objects:many-to-one association)
  • (4) The qualifications are replaced with one-to-many associations and composite identifiers in object classes on "many" side of one-to-many associations

We start from a case 1 where a multivalued attribute is transformed into into a class of objects and many-to-many association. The second case where a multivalued attribute is transformed into a class of objects and one-to-many-association is presented later.


Let us consider a class of objects: BOOK, and a multivalued attribute: author name. An attribute: author name in a class of objects: BOOK, is transformed in the following way.


An attribute: author name is promoted to a new class of objects: AUTHOR. Such transformation is commonly called as a "reification". Next, a single valued attribute: author name, is added as an identifier to a class: AUTHOR.


A multivalued attribute: author name is removed from a class of objects: BOOK. A many to many association: BOOK Is-written-by AUTHOR, is created between the classes of objects: BOOK and AUTHOR.


Multivalued attributes

  • Case 1: Transformation into a class of objects and many-to-many association

  • A multivalued attribute author-name in a class of objects BOOK is transformed in the following way:
    • An attribute author-name is promoted to a class of objects AUTHOR (such step is called as reification)
    • A single valued attribute author-name is added as an identifier to a class AUTHOR
    • A multivalued attribute author-name is removed from a class BOOK
    • A many-to-many association Is-written-by is created between the classes BOOK and AUTHOR

In the second step, we eliminate association classes and many to many associations through the transformations into one to many and many to one associations. An association class becomes and ordinary class of objects involved in both one-to-many and many-to-one associations.


For example. we consider a many to many association: STUDENTS Enrols SUBJECT, represented by an association class: ENROLMENT. We assume an optimistic case, a student enrols in a subject only one time. the pessimistic case, when a subject is enrolled by the same student many times is considered later.


First, an association class: ENROLMENT, is transformed in the following way. Many to many association: STUDENTS Enrols SUBJECT, is removed. A new one-to-many association: STUDENT Performs ENROLMENT, is added between a class of objects: STUDENT, and a class of objects: ENROLMENT.


Next, a one-to-many association: SUBJECT is-involved-in ENROLMENT, is added between a class of objects: SUBJECT and a class of objects: ENROLMENT.


A qualification with an attribute: subject code (code), is added on a student side of association: STUDENT Performs ENROLMENT and a qualification with an attribute: student number (s#), is added on a subject side of an association: SUBJECT is-involved-in ENROLMENT. The qualifications are need to provide information about a derived identifier of a class ENROLMENT.


In a general case, every many-to-many association can be replaced with one-to-many association CLASS OF OBJECTS (that represents an association) and many-to-one association. A derived identifier of CLASS OF OBJECT is a pair of identifiers taken from "one' sides of both associations. In our example it is a pair of attributes (student number, subject code).


Association classes

  • An association class Enrols is transformed in the following way:
    • Many-to-many association Enrols is removed
    • A one-to-many association Performs is added between a class STUDENT and a class ENROLMENT
    • A one-to-many association Is-involved-in is added between a class SUBJECT and a class ENROLMENT
    • A qualification with an attribute code is added on a STUDENT side of association Performs and a qualification with an attribute s# is added on SUBJECT side of association Is-involved-in

In step four of the logical design, we replace the qualifications.


In the present slide, a qualification with an attribute: room number (room#) is eliminated in the following way.


The attributes: building number (bldg#), and room number (room#), are copied to a class of objects: ROOM.


A pair of attributes that consists of building number (bldg#), and room number (room#), is tagged with IDn in a class of objects room where "n" depends on the total number of identifiers already determined in a class. Usually "n" is equal to 1 because its is why qualifications has been used: to determine an identifier fro a class that has no identification system.


In this way, a pair (bldg#, room#) becomes an identifier in a class of objects: ROOM.


An attribute: building number (bldg#), is tagged with FKn to denote a foreign key referencing an identifier building number (bldg#) in a class of objects: BUILDING.


A multiplicity of an association: BUILDING Has ROOM, on a ROOM side of qualified association is changed to * or 1..*.


It is because buildings have zero or more rooms.


At the end, a qualification with an attribute: room number (room#) is removed.


In a general case a qualification is removed through application of its definition. A qualification provides information'


about an identifier of a class, that has no own identification system, for example a class ROOM. Hence elimination of qualification must create an identifier for a class that has no own identification system. An identifier is created from the attributes used in qualification and from an identifier of a class located on a qualified side of association.


In our case, it is a class BUILDING with an identifier bldg#.


An identifier from a class on a qualified side of association (e.g. bldg#) becomes a future foreign key in a relational table ROOM. It is why it is tagged with FKn.

In the sixth step of logical design, one to one associations are transformed in the following way.


The object classes from both sides of one to one association are merged into a single class of objects.


For example a triple, that consists of the objects classes: CHAIRMAN, and SCHOOL, and an association: CHAIRMAN Is-head-of SCHOOL, is merged into one class of objects: SCHOOL. The attributes: employee number (emp#), first name, last name, and date of birth (dob), that come from an optional 0..1 side of the association: CHAIRMAN Is-head-of SCHOOL, are tagged with 0..1 tag as optional attributes.


An attribute like: name of school (name), that was an identifier of a class of objects on mandatory side, 1..1, of the association obtains a tag IDn.


A tag IDn means, that it is an identifier of a new class obtained from the merge of the classes CHAIRMAN and SCHOOL. It means that every instance of an object of the new class must have a value of an attribute school name (name). Later on IDn will be converted into PK, i.e. a primary key of a relational table implementing the new class of objects


An attribute like: employee number (emp#), that was an identifier of a class of objects on an optional side of association obtains a tag CKn. It is because it may happen that a school does not have a chairman due to an optional side of association on CHAIRMAN class side. Then a tag IDn is not appropriate because a value of an attribute employee number (emp#) maybe missing. In such a case and attribute employee number (emp#) can be at most a candidate key and not primary key.


Finally, one to one association is removed.


One-to-one associations

In the seventh step of logical design, the generalisation hierarchies are transformed with the superset, subset and association methods.


We start from a superset method.


A superset method transforms entire generalisation hierarchy into a single class of objects by moving all attributes to the immediate higher level class of objects, changing multiplicities of attributes, and adding categorisation attributes.


All attributes from the classes of objects at the lower levels of generalisation hierarchy are copied to the topmost level in the hierarchy and then such attributes become the optional attributes marked with 0..1 tag there.


For example, the attributes: project and thesis, are copied from the classes of objects: UNDERGRADUATE STUDENT and POSTGRADUATE STUDENT, to a class of objects: STUDENT.


A categorisation attribute, that determines a subtype of a superclass is added to a superclass.


For example, an attribute: type of student, is added to a class of objects: STUDENT.


Such attribute is necessary to distinguish between postgraduate and undergraduate students.


Generalizations - superset method

  • A superset method transforms entire generalization hierarchy into a single class of objects in the following way:
    • All attributes from the classes of objects at the lowest level of generalization hierarchy are copied to an immediate higher level and become optional attributes ([0..1] tag) there, e.g. the attributes project and thesis are copied from the classes UNDERGRADUATE-STUDENT and POSTGRADUATE-STUDENT to a class STUDENT
    • An attribute type-of-superclass is added to a superclass, e.g. and attribute type-of-students is added to a class STUDENT

After the attributes from lower levels of generalisation are moved to their parent class all classes below the parent class are removed from a conceptual schema.


The process is repeated until only the topmost class is left. It is important to remember that each time a set of attributes from subclasses is moved to a superclass we must remember about categorisation attribute. If a generalisation hierarchy has several levels then categorisation attributes from the lower levels must be moved to the higher levels. For example, of a class UNDERGRADUATE STUDENT has subclasses IT STUDENT and CS STUDENT then first elimination of the lowest level in the hierarchy contributes to a categorisation attribute it-or-cs. Then, elimination of the subclasses at the higher level contributes to a categorisation attribute pg-or-ug. Please remember that an attribute it-or-cs must be moved to the topmost class STUDENT as an optional attribute. i.e. it must be tagged with [0..1]. It is because we do not distinguish between it and cs students at postgraduate level.


Generalizations - superset method

    • All classes at the lowest level are removed
    • The steps above are repeated until only one class of objects is left

Next, to eliminate a generalisation hierarchy, we look at a subset method.


A subset method transforms the entire generalisation hierarchy, into several classes of objects in the following way. All attributes from the classes of objects at the higher levels of generalisation hierarchy, are copied to the classes of objects at the lowest levels of generalisation hierarchy.


For example, the attributes student number (s#), first name, last name and date of birth (dob) are copied from a class of objects: STUDENT, to the classes of objects: POSTGRADUATE STUDENT, and UNDERGRADUATE STUDENT.


Copying the attributes from a superclasses to subclasses is nothing else as application of inheritance. If a postgraduate student is a student then all attributes and associations of a student are inherited by a postgraduate students accordingly to a rule postgraduate student ISA student.

Finally, to eliminate a generalisation hierarchy, we look at an association method.


An association method transforms the entire generalisation hierarchy, into several of classes of objects in the following way. One of the identifiers from a superclass of objects is copied to all subclasses of objects located one level below in a hierarchy.


For example, an identifier: student number (s#) is copied from a class of objects: STUDENT, to the classes of objects: UNDERGRADUATE STUDENT, and POSTGRADUATE STUDENT. An identifier that has been copied obtains a tag FKn in the subclasses.


The same procedure is repeated for all subclasses of objects at the same level in a generalisation hierarchy.


After application of association method the classes STUDENT, UNDERGRADUATE STUDENT, and POSTGRADUATE STUDENT change their semantics/meaning. A class STUDENT contains the objects that are the common descriptions of postgraduate and undergraduate students. A class UNDERGRADUATE STUDENT contains the objects that are the descriptions of undergraduate students only. A class POSTGRADUATE STUDENT contains the objects that are the descriptions of postgraduate students only.


A meaning of an association is Consists_of. For example STUDENT Consists-of UNDERGRADUATE STUDENT description or STUDENT Consists-of POSTGRADUATE STUDENT description.
