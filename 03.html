Since late 1970s a large number of graphical notation has been proposed to formally (more or less) describe a conceptual schema of a database domain.


The first one was a notation of Entity-Relationship diagrams originally proposed as yet another database model. A graphical notation of Entity-Relationship diagrams (commonly called E-R diagrams) was published in 1976 by Peter Chen. in 1976 was too late compete with the relational database model and to convince large IT companies to drop their semi-implemented relational database systems and to switch to another data model. Luckily E-R diagrams appeared to be a prefect to for formal specification of database domains.


A database domain is represented in a notation of E-R diagrams by the entity types, attributes describing entity types, relationships between the entity types and identifiers of entity types.


In E-R diagram on the current slide, an entity type is visualised as a rectangle with the attached blobs labelled with attribute names. An entity type has a name located within a rectangle.


A relationship is visualised as a diamond and it is connected by straight lines to the rectangles representing the entity types. A relationship can be describe with the attributes in the same way as an entity type.


An attribute describing an entity type or a relationship is visualised as an oval with a name of attribute within an oval, and connected by a straight line to an entity type or to a relationship.


An E-R diagram visualised on the slide represents a database domain, where customers submit orders that consists of items. Items are produced by companies. Items are stored by the customers in their shopping carts. The orders are shipped to the customers.

A graphical notation of Object Modelling Technique diagrams (commonly called OMT diagrams) was developed in 1991 by Rumbaugh, Blaha, Premerlani, Eddy and Lorensen.


A notation of OMT diagrams consists of three types of models: object model, dynamic model and functional model.


An object model can be used for conceptual modelling. A database domain is represented in a notation of object model of OMT diagrams by the classes of objects, properties describing classes of objects, associations between the classes of objects and generalisation hierarchies.


A class of objects is visualised as rectangle, that contains the name of a class of objects, the names of properties and the names of methods (operations) applicable to a class of objects.


An association is visualised as a line connecting the classes of objects. An association has a name located close to a line and a direction indicator.


A generalisation is visualised as a small triangle with the lines, that link subclasses and super classes of objects.


An OMT diagram visualised on the slide represents a database domain, where the animators display drawable objects. There are two types of drawable objects: track segments and cars. Tracks consist of the track segments. Cars have the positions located on the track segments. The animators animate the animatees.


A notation of OMT diagram is quite similar to a notation of E-R diagrams. A class of objects is equivalent to an entity type, a property is equivalent to an attribute, an association is equivalent to a relationship. Both notations share a concept of generalization hierarchy.

A graphical notation of Unified Modelling Language (commonly called as UML diagrams), has been developed between 1994 and 1995 by Booch, Jacobson and Rumbaugh at Rational Software. UML consists of three types of diagrams: structure diagrams, behaviour diagrams and interaction diagrams.


Structure diagrams consist of component diagrams and object class diagrams.


A simplified notation of object class diagrams can be used for conceptual modelling.


A database domain is represented in a notation of simplified object class diagrams by the classes of objects, properties describing classes of objects, identifiers of the classes of objects, associations between the classes of objects, association classes, qualifications, and generalisation hierarchies.


A class of objects is visualised as a rectangle, that contains a name of a class of objects and the names of properties. A set of properties that can be used for identification purposes is denoted by ID tags.


An association is visualised as a line connecting the classes of objects. An association has a name located close to a line and a direction indicator.


A generalisation is visualised as a small triangle with the lines that link subclasses and super classes of objects.


A qualification is visualised as a rectangle attached to a class of objects and connected to an association.


A UML diagram visualised on the slide represents a database domain, where the customers submit orders, that consist of lines. A line contains a description of a part included in an order and supplied by a supplier. A part consists of other parts. A class of frequent customers is a subset of a class of customers. A line in an order is uniquely identified by an order number and line number.

A graphical notation of Object Role Modeling (commonly called as ORM diagrams), was developed in mid-1970s by Nijssen.


ORM diagrams can be used for conceptual modelling. A database domain is represented in ORM notation by the facts, objects involved in the facts, and roles played by the objects in the facts.


ORM notation is attribute free, unlike E-R notation or UML classes of objects notation.


A fact is a proposition, like James plays a football game today.


Ovals are used for visualisation of objects.


Pairs of adjacent rectangles are used for visualisation of roles played by the objects in facts.


An ORM diagram visualised on the slide represents a database domain, where the employees have employee numbers and occupy a room. An employee works for a department and has an extension number and access level code. An employee is contracted or tenured.

Now assume, that a person has no more than five email addresses and none or more mobile phone numbers. How do we represent such constraints imposed on the values of properties/attributes ?


The attributes like: email, and phone, are multivalued property/attributes. The complex values of multivalued properties/attributes are the sets of atomic values. The minimum and maximum total number of atomic values within a complex value of a multivalued property/attribute, is called as multiplicity of an property/attribute.


A suffix like [ 1 .. 5 ] within the square brackets, at the end of the property/attribute name indicates a multivalued property/attribute and means, that multiplicity of a property/attribute is from 1 to 5. A suffix [ 1 .. 5 ] means, that a complex value of a multivalued property/attribute, must have at least one atomic value and no more than 5 atomic values.


A suffix like [ * ] or [ 0 .. * ] denotes a complex value, that consists of zero or more atomic values.


A suffix like [ 1 .. * ] means a complex value, that consists of at least one or more atomic values.


A suffix like [ m .. n ] where m < n means a complex value that consists of at least m atomic values and at most n atomic values.


Default multiplicity of a property/attribute is [ 1 ].


It means that all properties/attributes, that do not have a suffix, like for example, social security number, name, date of birth, must have a precisely one value.


(Simplified) Class of Objects

  • A person has from one to five email addreses and zero or more phone numbers

  • Multiplicity of attribute like [1..5] (from one to five), zero or more ([*] or [0..*]), one or more ([1..*]), optional (zero or one, [0..1]), from "m" to "n" ([m..n]) follows a name of attribute
  • Default multiplicity is "one" ([1])

Assume, that in our database domain, a person is identified by a social security number, and independently by a triple of the properties/attributes that consists of a name, date of birth, and address.


Such identification means, that in this case no two people a database domain can have the same social security number and the same triple of values of the properties/attributes name, date of birth and address. It is possible that two or more people have the same date of birth and the same name, it is possible that two or more people have the same name and date of birth, and it is also possible two or more people have the same name and the same address. However, it is not possible that two or more people have the same name, date of birth and address.


A tag ID followed by a digit attached to name of a property/ attribute means, that such property/attribute is an identifier of all instances of objects that belong to a given class.


For example, a tag ID1 following a name of a property/attribute social security number, means that a social security number is an identifier. Indeed, in the reality, all people have different social security numbers.


If the same tag, for example ID2 follows the names of several properties/attributes then, it means that an identifier consists of several properties/attributes each marked with a tag ID2. If an identifier consists of several attributes, then it means that each object in a class is identified by a tuple of values of such properties/attributes. An identifier, that consists of several properties/attributes is called as a composite identifier.


For example, a triple of properties/attributes (name, data of birth, address) is a single composite identifier.


It is important to note, that when we find the identifiers fro the classes of objects we also try to find minimal identifiers, i.e. the identifiers that consists of the smallest number of properties/attributes. A social security number if a minimal identifier, because we cannot remove any property/attribute from such identifier and still have a valid identifier. A triple of properties/attributes ((name, data of birth, address) is a minimal identifier because as we mentioned earlier any proper subset of of two elements of the triple is not an identifier. A pair of properties/attributes (social security number, date of birth) is not a minimal identifier because a date of birth can be removed from is and still social security number alone is an identifier.


(Simplified) Class of Objects

  • A person is identified by a social security number and independently by a triple (name, date of birth, address)

  • A tag IDx following a name of attribute denotes an identifier
  • If the same tag (e.g. ID2) follows the names of several attributes then it means that an identifier consists of several attributes
  • If an identifier consists of several attributes then it means, that each object in a class is identified by a tuple of values of the attributes
  • An identifier which consists of several attributes is called as a composite identifer

Now, we present a short summary of how to create a class of objects, and how to provide properties/attribute to describe a class of objects. We also look at the different types of properties/attributes.


A class of objects represents a set of homogeneous real world objects, described by the same set of properties/attributes, and related to other classes of objects by the same set of associations.


A class of objects is visualised as a single column table with a name of a class in a header of the table and a list of property/attribute names below a header.


[ m .. n] tags following the names of properties/attributes indicate a multiplicity of a property/attribute. If a property/attribute does not have [ m..n ] tag the its default multiplicity is [1..1]. It means, that it is a single valued mandatory property/attribute and its multiplicity is equal to 1.


[ 0 .. 1 ] tags following the names of properties/attributes indicate that such attributes are optional attributes. It means that a property/attribute may have a value or not. It is still a single valued property/attribute.


ID followed by a digit tags attached to the names of properties/attributes indicate, that such properties/attributes are the identifiers, or they are the members of identifiers. if more than one property/attribute is followed by ID digit tag then it denotes a composite identifier that consists of several properties/attributes.


A forward slash preceding a name of property/attribute indicates, that such property/attribute is a derived attribute. It means that value of such attribute can be derived/computes from the values of one of more properties/attributes describing a class of objects.

The next component of a notation of simplified UML class diagram is an association. Let us assume, that a database is supposed to contain information about peoples, vehicles, and owners of vehicles. To represent people and vehicles, we create the classes of objects: PERSON and VEHICLE. The same classes of objects have already been created, and described in the previous presentation.


A fact, that a person owns a car, can be represented by an abstract link between an instance of a class person and an instance of a class car. When the instances of the objects are represented as the icons on a computer screen, then an abstract link is a line connecting two icons. In this case it is an icon of a person and an icon of a car. A collection of the homogeneous links between the objects, is called an association.


An association is visualised as a solid line that connects the visualisation of two classes of objects. A name above an association line is an association name. In our case association name is, Owns, and the association can be interpreted as PERSON Owns VEHICLE. A direction of interpretation is determined by a small solid triangle, that follows a name of an association. So, in this case, it is definitely not, VEHICLE Owns PERSON.


A star symbol located on the right hand side of association, is a multiplicity of association, and in this case, it means that a person owns zero or more cars.


A default value of multiplicity is one. It also means that a car has only one owner and, that each car must have an owner.


Multiplicities of associations will be more thoroughly discussed later in this presentation.

Now, we look at an example of a conceptual schema, that represents a database domain, where companies employ employees. A company employs many employees, and an employee is employed by zero or more companies. It is also possible, that a company has no (zero) employees. It is an example of one to many association, or many to one, well, it really does not matter, because in both cases, it is the same association. One to many means, that multiplicity of an association on one side is one, or, zero or one, and multiplicity of the same association on the other side is many, or zero or many.


An instance diagram below a conceptual schema shows the instances objects in a class company, visualised as the small black ovals and the instances of objects in a class employee, also visualised as the small black ovals. The abstract links between the instances of companies, and the instances of employees, represent information who is employed by which company. The abstract links are visualised in the instance diagram, as straight lines linking the black ovals representing the instances of objects. It is clearly visible that multiplicity of an association, employs, is one to many because the instances of an object in a class company, are linked to none or many instances of an object in a class employee. And each instance of an object in a class employee, is linked at most to one instance of an object in a class company.


It is possible that a company does not employ any employees and some employees, hopefully temporary, are not employed by any company. The answers to the questions why a person is called as an Employee even if he/she is not employed and how it is possible that a company does not employ anyone depends on our view of the reality. In some situations it is possible that an employee moves from one company to another and such transfer happens in a certain no zero period of time. In such a case it is possible to say that a person is still an employe because his/her employment with another company is guaranteed in a moment when information is recorded in a database. In the sam way w may argue that a completely new company in a moment when it is registered does not really have any employees, however, it is registration can be sued as a proof of its existence. So, minimal multiplicity very frequently depends on our subjective view of the reality.

In this slide, we look at a sample conceptual schema, that represents a database domain where students enrol in subjects. The conceptual schema consists of two classes of objects, student and, subject, linked with an association enrols. A multiplicity of association enrols is, many to many, because a students is allowed to enrol in many subjects and a subject can be enrolled by many students.


We would like to add to a conceptual schema, information about enrolment date of each enrolment, performed by a student. If we add an attribute enrolment date to a class student, then such attribute must be a multivalued attribute, because a student can enrol in many subjects on many different days. Additionally, we are not able to represent information what subject was enrolled on what date, because a conceptual schema does not represent the links between the values of multivalued attributes, and the instance of objects in another class.


An idea to add an attribute, enrolment date, to a class subject, fails because of the same reasons. The attribute becomes multivalued attribute, because a subject may have many enrolments. Additionally, the values of the attribute cannot be associated with the objects from a class student. A correct solution to the problem is to add an attribute, enrolment date, as a link attribute.


A link attribute is an attribute that describes an association. In our case, an association, student enrols in a subject, is described by a link attribute, enrolment date. Such solution is correct, because enrolment fact is related to both object classes, student and subject. An enrolment in a subject is performed by a student on a given day. It means that enrolment is described by enrolment date.


A link attribute is visualised as a "class-like" rectangular box without a name of class in a header connected with a dashed line to a line representing an association.

A class of objects is a collection of homogeneous instances of objects, described by almost the same set of attributes. If we consider the links as objects, then a collection of homogeneous links, described by almost the same set of attributes becomes a class of objects. Such class of objects consists of a collection of objects (links) that represents an association between two classes of objects. This is why, a class of objects that consists of links, is called as, an association class. It is correct to say, that an association class is a class that represents an association.


Let‚Äôs consider a database domain, where students enrol in subjects. Such domain can be represented by a conceptual schema, where the classes, student and subject, are linked with an association, enrols. If we would like to consider the links between students and subjects as objects, then a collection of such links becomes an association class called as enrolment. If we record a fact that enrolment is performed on a given day, then the association class is described by an attribute enrolment date.


An association class enrolment is graphically represented in the same way as a class of objects with a dashed line connecting a class box with an association and a name of a class in a header. An association class, is the same class as other classes of objects, and it can be involved in the associations with other classes of objects. It is also possible to upgrade a link attribute to an association class. To do so, we apply a procedure of reification, that transforms an attribute into a class of objects.In our case a link attribute or a group of link attributes is promoted into a class of objects, enrolment.

An interesting problem related to an association classes, is a question about an identifier of association class. From where, do we get an identifier of an association class ?


An association class may have its own identifiers, for example, an attribute enrolment number in a class enrolment.


An association class always has one or more default and composite identifier. A default identifier is created in a process similar to deriving an identifier using a qualification (explained later in this presentation). An association class, that represents many to many association has a default identifier, that consists of the identifiers of classes from both sides of the association.


For example, in the present slide a pair of attributes: (student number,subject code), is a default composite identifier of a class enrolment. A default identifier is created by concatenation of an identifier from a class student (student number), and identifier from a class subject, (subject code).


Another default identifier can be also created from a pair of attributes: (student number, subject title), can also be used as an identifier of a class enrolment.


It is possible to always create a default identifier for an association class and because of that there is no need to include an artificial identifier, like enrolment number, into a description of a class of objects.


In what circumstances a pair of identifiers taken from the classes of objects linked by an association is an identifier of the respective association class ? Assume, that each pair of object instances is linked by at most one link. Hence, a pair of identifiers of the linked object instances is completely sufficient to identify each link. However, sometimes two instances of object can be linked by more than one link. For example a student enrols a subject and participation in the subject is unsuccessful and the student must enrol the subject one more time. It is so called "pessimistic view of the reality" ;). Then, two object instances are connected by two links and a pair of identifiers of the objects is not sufficient to distinguish between the links. Then, association must be qualified by one or more additional attributes. A concept of qualification will be discussed later in this presentation.


Association Class

  • What about an identifier of an association class ?
  • An association class may have its own identifiers, for example enrolment-number in a class ENROLMENT
  • An association class that represents many-to-many association has a default identifier that consists of identifiers of the classes from both sides of association, for example a pair (snumber, code) in a class ENROLMENT

  • There is no need to include a default identifier into a description of a class of objects

Let us consider a database domain of a university campus, where buildings are located at campus, and rooms are located within each building. A building at a university campus is described by a unique building number, optional building name and the total number of floors. A room is described by a room number and an area. A building consists of rooms. A conceptual schema of the database domain is visible in the middle of the present slide. if someone would tell you that a lecture class in a subject is in room 125 then you would says: all right but which building are you tallking about ?


A building is identified by an attribute building number (bnumber attribute in a class building). What is an identifier of a class room ? An attribute room number (rnumber attribute in a class room) is not an identifier, because many different buildings may have the rooms with the same numbers. Is it worth to create, and to use an artificial identifier, like for example roomid ?n Such identifier is completely useless in practice, because it does not provide information which building hosts a room with a given roomid. For example, it would be hard to find a location of a room 546 because such number does not provide any information about the building number, and searching for a room by traversing the buildings one by one would be too time consuming.


In the reality, an identifier of a class room is a composite identifier, that consists of the attributes building number from a class building, and room number from a class room. It is why, when someone says, that a lecture class in in a room 125 then you ask which building is it ?


Unfortunately, we cannot add an attribute building number to a description of a class room. It is because in conceptual modelling an attribute can be used fro a description of one class only. So, how do we represent the composite identifiers that consist of the attributes from more than one class ?

A qualification is visualised in a simplified class diagram as a rectangular box, that contains one or more attribute names. A box with the attribute names is on one side attached to a class of objects, and on the other side to an association. The other side of association is attached to a class of objects, that does not have its own identification system, for example a class of objects room.


A qualification of a class building, and association consists of, with an attribute room number (rnumber attribute) means, that in each building there is at most one room with a given number.


Please, note 0 ..1 (optional) multiplicity of an association "Consists of" on its right hand side. It means, that for example, there is a most one room number 17 in a building number 35. Such rule applies to all buildings on a campus and to all rooms in the buildings. Inn the other words, we can say, that an attribute, room number, is a local identifier in a class room, because all rooms in a given building must have different numbers. Saying it in yet another way, if we pick a building and we pick a room number than there is at most one room with such number in the building.


It also means that a pair of attributes building number and room number, is a global identifier of a class, room. This is how an identifier of a class, room is created from concatenation of identifier of a class building and an attribute taken from a class, room, and included in a qualification box. The names of attributes located within a qualification box, together with an identifier of class the box is attached to, become an identifier of a class on the other side of qualified association.


This is why a pair of attributes room number and building number is an identifier of a class room. In a general case, qualification should be used in a situation, when a class of objects does not have its own identification system, like in a case of rooms located in university buildings.

As we discussed this problem earlier, sometime a pait of identifier from the classes of obejcts on both sides of many-to-many association visible as an association class is not sufficient to be a default identifier. Double qualification is graphically a bit awkward because it requires replacement of many-to-many association and association class with one-to-many association, a class of objects the same as association class and many-to-one association. To simplify a graphical notation we make it possible to represent it a qualification of the middle part of many-to-many association.
